INVERSE SCIENCE METHOD
Document-only guidance for governed problem solving and future Agent_X evolution agents

Version: 2.0
Status: Advisory method document, final release for document-only Agent_X integration
Runtime status: Not part of the required Agent_X seed-kernel runtime path

Revision note:
    Version 2.0 is the final document-only release. It makes no substantive expansion to
    the method. It preserves inverse science as advisory guidance, confirms that no runtime
    integration should be inferred from this document, and treats future edits as valid only
    when a concrete defect, ambiguity, architectural drift, repeated misuse, or proven
    compression need is observed.


============================================================
1. PURPOSE
============================================================

Inverse science is a disciplined method for deciding what input, action, design, prompt,
parameter, tool call, code change, or strategy to try next when the desired output is
known but the path to that output is uncertain.

Use inverse science when the central question is:

    What should be tried next to produce, approach, explain, or improve a desired output?

The method is especially useful when the system being acted on behaves like a black box:
the agent can choose inputs and observe outputs, but it does not fully know the hidden
mechanism connecting them.

This document is intended as guidance for an external evolution agent, coding agent,
research agent, or planning agent that reads the Agent_X repository and decides how to
improve, specialize, or operate it.

It is not a mandate to add a new runtime subsystem.


============================================================
1A. INTENDED USE AND NON-GOALS
============================================================

This document is meant to be read by an external agent before it proposes changes,
experiments, patches, or strategy updates.

It is intended to guide:

    - what problem to define,
    - what output to seek,
    - what candidate action to try,
    - what evidence to collect,
    - when to stop,
    - when to reframe,
    - and how to avoid mistaking activity for progress.

It is not intended to be:

    - a mandatory runtime loop,
    - a replacement for governance,
    - a permission system,
    - a self-modification engine,
    - a Bayesian optimization framework,
    - a full scientific-method textbook,
    - or a justification for adding machinery before evidence shows it is needed.

For Agent_X, this document should help an external evolution agent reason better while
leaving the seed kernel small and method-neutral.


============================================================
2. KERNEL BOUNDARY
============================================================

Inverse science is a reasoning doctrine, not a required kernel primitive.

For Agent_X, this document must be treated as advisory unless a future profile explicitly
adopts it.

The seed kernel should remain:

    - small
    - governed
    - replayable
    - checkpointable
    - profile-neutral
    - method-neutral
    - tool-mediated
    - evaluation-aware
    - safe to evolve by an external agent

Do not add inverse-science runtime machinery to the seed unless the change clearly improves
the kernel without weakening minimality, governance, replay, or extensibility.

The correct relationship is:

    Agent_X core:
        governed execution kernel

    Inverse science:
        optional method for choosing better next actions

    External evolution agent:
        may read this document and use it to decide what code, tests, profiles, or
        documentation to change next


============================================================
2A. DOCUMENT-ONLY INTEGRATION RULE
============================================================

This method should be integrated into Agent_X as a document first.

The external evolution agent may read it and use it to decide what to try next, but the
seed kernel must not be required to execute inverse science internally.

Document-only integration is preferred when:

    - the seed kernel is still being stabilized,
    - the method is useful but not required for every agent type,
    - runtime implementation would add complexity before there is evidence of need,
    - future agents can apply the method through ordinary planning, testing, and review,
    - the project needs doctrine more than machinery.

Runtime integration may be considered later only if:

    - a profile explicitly needs inverse-science behavior,
    - the core runtime can remain profile-neutral,
    - governance-before-action is preserved,
    - replay and checkpoint behavior remain testable,
    - the implementation is small, typed, and replaceable through ports,
    - there is evidence that document-only use is insufficient.

Until those conditions are met, inverse science is guidance for the agent, not a kernel
subsystem.


============================================================
2B. DOCTRINE, PROFILE, AND RUNTIME DECISION GATE
============================================================

Before integrating any part of inverse science, classify the proposed integration level.

Doctrine level:
    A document, checklist, rubric, or acceptance criterion. This is the default and safest
    form for the seed stage.

Profile level:
    A profile or agent mode that chooses to use inverse-science behavior while the core
    remains neutral. This is acceptable only when the profile can be added without changing
    the governed runtime path.

Runtime level:
    Core code that executes inverse-science stages directly. This should be avoided unless
    there is strong evidence that doctrine-level and profile-level use are insufficient.

Decision rule:

    IF the improvement can be achieved as doctrine, use doctrine.
    ELSE IF it can be achieved as profile behavior, use profile behavior.
    ELSE IF runtime machinery is unavoidable, keep it small, port-based, governed,
    replayable, and replaceable.

The burden of proof is always on the proposal that adds runtime complexity.


============================================================
2C. INTEGRATION-PERMISSION TEST
============================================================

Before an external agent proposes any repository change based on this document, it must
classify the change by permission level.

Allowed by default:

    - documentation improvements,
    - test additions that do not alter runtime behavior,
    - examples, rubrics, acceptance criteria,
    - profile-level guidance that does not change the core runtime path,
    - small contract clarifications that preserve existing behavior.

Requires stronger evidence:

    - new runtime branches,
    - new required planner logic,
    - new persistent state fields,
    - new evaluation gates that can block normal operation,
    - new tool policies,
    - new abstractions imported by the seed runtime.

Reject unless explicitly justified:

    - direct tool execution outside governance,
    - runtime self-modification,
    - large optimization frameworks,
    - heavy dependencies,
    - method-specific code embedded in the neutral kernel,
    - changes that make replay or checkpointing harder to test.

Default rule:

    If the same benefit can be achieved by documentation, tests, or profile guidance, do not
    add runtime machinery.

============================================================
2F. ANTI-BLOAT DECISION RULE
============================================================

Before proposing any implementation inspired by this document, apply this rule:

    IF the benefit can be achieved by documentation, do documentation.
    ELSE IF the benefit can be achieved by tests, do tests.
    ELSE IF the benefit can be achieved by a profile, do profile-level behavior.
    ELSE IF the benefit requires runtime code, keep the change minimal, port-based,
    governed, typed, replayable, and removable.

Reject the change if it adds machinery whose value cannot be measured by a clearer target,
a better test, a safer decision, a more replayable trace, or a smaller future evolution step.

The seed should become easier to evolve after the change, not merely more capable in one
narrow scenario.

Adoption rule:

    - add the method as documentation first,
    - mention it as optional guidance,
    - do not import it into runtime code,
    - do not create new ports or state fields only for this method,
    - do not add dependencies for inverse-search machinery,
    - revisit runtime adoption only after repeated external-agent use shows a real need.

A correct repository integration is successful when a future agent can read the document,
apply the method, and produce better governed patches without the kernel becoming larger.

This document should stay useful as doctrine, not grow into an informal runtime spec.

Future edits should follow these rules:

============================================================
3. BASIC DEFINITION
============================================================

Normal science often starts with a hypothesis and tests whether the world agrees.

Inverse science starts with a desired output and searches for the input that can produce it.

Forward question:

    Given input X, what output Y occurs?

Inverse question:

    Given desired output Y, what input X should be tried?

Examples:

    Desired output:
        Better kernel replayability

    Candidate inputs:
        Add replay tests
        Simplify event schema
        Tighten checkpoint contracts
        Remove non-deterministic runtime behavior

    Observed outputs:
        Tests pass or fail
        Replay succeeds or diverges
        Trace becomes clearer or noisier
        Kernel becomes simpler or more complex

Inverse science turns improvement into a disciplined search loop rather than an
unstructured sequence of guesses.


============================================================
4. CORE LOOP
============================================================

The core loop is:

    1. Define desired output.
    2. Define hard constraints and soft preferences.
    3. Define the input domain.
    4. Characterize the black box.
    5. Assess whether the target is identifiable and feasible.
    6. Build or choose a provisional model.
    7. Form an inverse hypothesis.
    8. Generate candidate inputs.
    9. Rank candidates by expected value, information gain, risk, cost, and novelty.
    10. Route any probe through governance.
    11. Execute only approved probes.
    12. Observe and record the output.
    13. Analyze the result.
    14. Update the model, memory, best-known solution, and negative knowledge.
    15. Decide whether to continue, backtrack, report, or stop.

Compact form:

    desired output
    -> constraints
    -> input domain
    -> black-box characterization
    -> candidate inputs
    -> governed probe
    -> observed output
    -> analysis
    -> memory update
    -> next candidate or stop


============================================================
5. WHEN TO USE INVERSE SCIENCE
============================================================

Use this method for:

    - black-box optimization
    - prompt improvement
    - codebase evolution
    - tool strategy selection
    - scientific search
    - debugging uncertain systems
    - architecture improvement
    - product strategy search
    - parameter tuning
    - experiment planning
    - agent behavior refinement
    - multi-step real-world problem solving

Use it when:

    - the target is clearer than the path
    - trial and observation are possible
    - mistakes can teach useful negative knowledge
    - candidate actions can be compared
    - uncertainty should be reduced systematically
    - governance is needed before acting

Do not force it for:

    - trivial tasks
    - direct factual lookup
    - simple deterministic transformations
    - tasks where the next action is already obvious
    - tasks where trial is impossible, unsafe, or unethical
    - tasks where no meaningful output can be observed


============================================================
6. REQUIRED DISCIPLINE
============================================================

An agent using inverse science must follow these rules:

    - Define the target before acting.
    - Separate hard constraints from soft preferences.
    - Define what counts as success, partial success, failure, and uncertainty.
    - Define what inputs are allowed and forbidden.
    - Do not confuse the best-found input with a proven optimum.
    - Do not overclaim results.
    - Record failed attempts as useful negative knowledge.
    - Preserve the reason each candidate was chosen.
    - Route all real probes through governance.
    - Avoid irreversible or unsafe actions unless explicitly authorized and justified.
    - Checkpoint before and after important probes.
    - Preserve replayability.
    - Prefer small informative probes over large uncontrolled changes.
    - Stop when further probing is unjustified by cost, risk, or diminishing returns.


============================================================
7. BLACK-BOX MODEL
============================================================

A black box is any system where the agent can observe some inputs and outputs but does not
fully know the mechanism.

For Agent_X evolution, the repository can be treated as a black box:

    Input:
        A code change, documentation change, test addition, profile change, contract change,
        or architecture simplification.

    Output:
        Test results, trace behavior, replay behavior, complexity change, governance behavior,
        maintainability, and seed-kernel score.

The black-box characterization should record:

    - observable inputs
    - observable outputs
    - hidden-state hypotheses
    - controllable variables
    - uncontrollable variables
    - delays between input and output
    - noise or randomness
    - nonstationarity or drift
    - coupled subsystems
    - safety limits
    - measurement limits
    - likely failure modes


============================================================
8. TARGET DEFINITION
============================================================

Before selecting any candidate action, define the desired output.

A target definition should include:

    - target name
    - target description
    - success tolerance
    - unacceptable outputs
    - measurable indicators
    - evidence required
    - uncertainty that can be tolerated
    - uncertainty that must be reduced

Example for Agent_X:

    Target:
        Improve Agent_X as a minimal governed universal seed kernel.

    Desired output:
        A smaller, cleaner, more replayable, more testable kernel that can be safely evolved
        by an external coding agent into many specialized agents.

    Hard constraints:
        Do not add runtime bloat.
        Do not bypass governance.
        Do not introduce direct unmediated tool execution.
        Do not break the public entrypoint.
        Do not weaken checkpoint or replay behavior.

    Soft preferences:
        Prefer simpler code.
        Prefer clearer contracts.
        Prefer stronger tests.
        Prefer less duplication.
        Prefer profile-based specialization over core changes.


============================================================
9. CONSTRAINTS
============================================================

Hard constraints are non-negotiable. A candidate that violates them must be rejected.

Soft constraints are preferences or tradeoffs. A candidate may violate a soft constraint only
if the expected gain is explicit and justified.

For each problem, define:

    Hard constraints:
        - safety limits
        - governance limits
        - ethical limits
        - legal or regulatory limits
        - runtime invariants
        - input-domain boundaries
        - budget limits
        - irreversible-action limits

    Soft constraints:
        - cost
        - elegance
        - simplicity
        - speed
        - maintainability
        - coverage
        - explainability
        - extensibility

The agent must not treat soft preferences as hard constraints unless explicitly stated.


============================================================
9A. MINIMUM EVIDENCE STANDARD
============================================================

Every nontrivial claim of improvement must identify its evidence class.

Evidence classes:

    unsupported_claim:
        A statement with no test, trace, inspection, or measurable observation. This should
        not be treated as progress.

    inspection_evidence:
        A human or agent inspected the artifact and found a plausible improvement. Useful,
        but weaker than executable evidence.

    test_evidence:
        A relevant test, fixture, replay, or check passed after the candidate change.

    comparative_evidence:
        The candidate was compared against a previous state, baseline, or alternative.

    replicated_evidence:
        The result was reproduced across repeated runs, examples, or conditions.

    negative_evidence:
        The candidate failed, exposed a weakness, or ruled out an assumption in a way that
        improves future decisions.

For Agent_X evolution, strong evidence usually requires at least one of:

    - a passing test that did not exist before,
    - a replay or checkpoint fixture,
    - a reduced or simplified runtime path,
    - a clearer contract boundary,
    - a documented removal of ambiguity,
    - or a recorded negative result that prevents repeated bad patches.

A change that only sounds better but produces no observable evidence should be treated as
low-confidence.


============================================================
9B. OBJECTIVE AND EVIDENCE SCORING
============================================================

The agent should not rely on vague impressions of improvement.

For each inverse-science task, define a scoring structure before ranking candidates.

Minimum score fields:

    - target_distance:
        How far the observed output is from the desired output. Lower is better.

    - hard_constraint_status:
        Whether all hard constraints were preserved. Any hard failure blocks success.

    - soft_preference_score:
        How well the candidate satisfies preferences such as simplicity, maintainability,
        speed, clarity, or extensibility.

    - information_gain:
        How much the probe reduced uncertainty, even if it did not directly improve the
        result.

    - risk_cost_score:
        Combined cost, danger, irreversibility, runtime complexity, or maintenance burden.

    - evidence_strength:
        How strong the evidence is that the observed effect is real, reproducible, and
        attributable to the candidate.

A candidate should be considered useful if it does at least one of the following without
violating hard constraints:

    - improves the best-known solution,
    - reduces uncertainty,
    - identifies a bad region,
    - falsifies a weak hypothesis,
    - exposes a hidden constraint,
    - improves the model used for future decisions,
    - justifies stopping or reframing.


============================================================
9C. EVIDENCE THRESHOLD BY CHANGE TYPE
============================================================

Not every change requires the same evidence level. Use the smallest sufficient evidence
standard, but do not accept unsupported improvement claims.

Documentation-only change:
    Minimum evidence: inspection_evidence.
    Stronger evidence: the document removes ambiguity, adds an acceptance criterion, or
    prevents a known bad implementation path.

Test-only change:
    Minimum evidence: test_evidence.
    Stronger evidence: the test fails before the relevant fix or guards a known invariant.

Contract/schema change:
    Minimum evidence: inspection_evidence plus test_evidence.
    Stronger evidence: replay/checkpoint compatibility is demonstrated.

Runtime-path change:
    Minimum evidence: test_evidence plus comparative_evidence.
    Stronger evidence: replay evidence, governance evidence, and no increase in unnecessary
    runtime complexity.

Tool/governance change:
    Minimum evidence: explicit safety rationale plus test_evidence.
    Stronger evidence: blocked-action and allowed-action cases are both demonstrated.

External evolution agents should prefer changes with low implementation risk and high
evidence value before attempting broad architectural changes.


============================================================
10. INPUT DOMAIN
============================================================

The input domain is the set of actions or changes the agent is allowed to try.

For software evolution, candidate inputs may include:

    - add a test
    - remove dead code
    - tighten a contract
    - simplify a runtime path
    - add a profile
    - improve a document
    - add a replay fixture
    - improve error handling
    - refactor an adapter
    - add an evaluator hook
    - remove duplicate logic

For each input, record:

    - what will be changed
    - why it is allowed
    - expected output
    - expected information gain
    - expected risk
    - expected cost
    - rollback path
    - evaluation method


============================================================
11. INVERSE-PROBLEM TYPES
============================================================

Classify the inverse problem before choosing a strategy.

Common types:

    Deterministic inverse:
        The same input reliably produces the same output.

    Stochastic inverse:
        The same input may produce different outputs.

    Optimization problem:
        The goal is to improve a score rather than hit an exact output.

    Constraint-satisfaction problem:
        The goal is to find any input satisfying all hard constraints.

    Multiobjective problem:
        The goal involves tradeoffs among competing outputs.

    Ill-posed inverse problem:
        Multiple inputs may explain the same output, or small input changes may cause large
        output changes.

    Many-to-one inverse:
        Many valid inputs can produce the desired output.

    Nonstationary black box:
        The system changes over time, so old observations may become less reliable.

    Expensive-trial problem:
        Each probe has high cost, time, or risk.

    Unsafe-trial problem:
        Some probes could cause unacceptable harm or irreversible effects.

The search strategy should depend on this classification.


============================================================
12. IDENTIFIABILITY
============================================================

Do not begin optimization until the target is at least provisionally identifiable.

Ask:

    - Can the desired output be observed?
    - Can the relevant inputs be controlled?
    - Are there hidden variables that dominate the output?
    - Are measurements precise enough?
    - Is the target meaningfully different from failure?
    - Can trial results distinguish among candidate explanations?
    - Is the problem underdetermined?
    - Could many inputs produce indistinguishable outputs?

Possible identifiability statuses:

    identifiable:
        Continue with candidate generation and ranking.

    partially identifiable:
        Continue, but prefer high-information probes and report uncertainty.

    not identifiable:
        Reframe the problem, improve observability, or stop.

    unknown:
        Perform characterization probes before optimization.


============================================================
13. MODEL SET
============================================================

The agent should maintain a provisional model set, not assume one model is correct.

Possible model families:

    - constraint-only model
    - lookup-table model
    - nearest-neighbor model
    - local-linear model
    - rule-based causal model
    - probabilistic surrogate model
    - tree-based surrogate
    - mechanistic model
    - subsystem model
    - ensemble model

The active model should be chosen by:

    - consistency with observations
    - usefulness for inverse search
    - robustness to noise
    - compatibility with constraints
    - computational cost
    - explainability
    - ability to suggest informative next probes

The agent should keep alternative models alive when uncertainty is high.


============================================================
14. INVERSE HYPOTHESIS
============================================================

An inverse hypothesis is a tentative claim about what input regions may move the output
toward the target.

It should state:

    - which inputs are likely useful
    - which variables matter most
    - which unknowns most need resolution
    - what evidence would support it
    - what evidence would weaken or falsify it
    - what candidate probes follow from it

Example:

    Hypothesis:
        The kernel score will improve more from tightening replay/checkpoint tests than
        from adding new runtime capabilities.

    Candidate probes:
        Add replay invariant tests.
        Add checkpoint round-trip tests.
        Remove runtime paths not covered by replay.
        Strengthen trace expectations.

    Falsification signal:
        Tests show replay is already strong, while governance or contract boundaries remain weak.


============================================================
15. CANDIDATE GENERATION
============================================================

Generate candidates from multiple sources:

    - literature or prior work
    - current model
    - inverse hypothesis
    - boundary cases
    - conservative safe probes
    - high-information probes
    - local refinement around a good result
    - global exploration of untested regions
    - failed-region repair
    - replication of suspicious results

A candidate should not be just an action. It should include a reason.

Candidate record:

    - candidate id
    - input/action/change
    - source
    - expected output
    - expected information gain
    - expected risk
    - expected cost
    - hard-constraint check
    - rollback plan
    - rationale


============================================================
16. CANDIDATE RANKING
============================================================

Do not rank candidates by intuition alone.

Rank candidates using an acquisition score that considers:

    - expected closeness to target
    - expected information gain
    - constraint satisfaction
    - safety risk
    - cost
    - novelty
    - reversibility
    - uncertainty reduction
    - chance of improving the best-known solution

A practical scoring rule:

    acquisition_score =
        expected_target_gain
        + expected_information_gain
        + novelty
        + reversibility_bonus
        - constraint_risk
        - safety_risk
        - cost
        - complexity_penalty

The highest-ranked candidate is not automatically allowed. It must still pass governance.


============================================================
16A. PATCH GRANULARITY AND VARIABLE CONTROL
============================================================

When using inverse science for codebase evolution, control the number of variables changed
at once.

Prefer one primary change per probe:

    - one contract tightened,
    - one test family added,
    - one runtime path simplified,
    - one document clarified,
    - one profile boundary improved,
    - or one obsolete component removed.

A multi-change patch is allowed only when the changes are inseparable and the report states
why they must be evaluated together.

Variable-control rule:

    IF a candidate changes too many things at once,
    THEN split it into smaller candidates unless splitting would destroy the evidence value.

This prevents the agent from producing a large patch where the cause of improvement or
regression cannot be identified.


============================================================
17. EXPLORATION AND EXPLOITATION
============================================================

Use exploration when:

    - little is known
    - model uncertainty is high
    - recent candidates have failed
    - the search is stuck
    - identifiability is weak
    - many regions remain untested

Use exploitation when:

    - a promising region has been found
    - the target is nearly reached
    - uncertainty is low enough
    - additional local refinement is cheap and safe

A simple policy:

    - early stage: mostly exploration
    - middle stage: balanced exploration and exploitation
    - near success: mostly exploitation
    - after stagnation: return to exploration or reframe


============================================================
18. GOVERNED PROBING
============================================================

A probe is any action that tests a candidate input against the black box.

For Agent_X, probes may include:

    - running tests
    - modifying code
    - creating a new profile
    - changing a contract
    - removing a module
    - adding documentation
    - checking replay
    - checking trace output
    - evaluating seed-kernel score

Every probe must be governed.

Before execution, check:

    - Is the candidate within the allowed input domain?
    - Does it violate any hard constraint?
    - Is it reversible?
    - Does it require user approval?
    - Does it bypass tool mediation?
    - Does it weaken safety?
    - Does it create runtime bloat?
    - Does it preserve replayability?
    - Is the expected information gain worth the cost?


Governance outcome types:

    allow:
        The candidate may be executed as proposed.

    allow_with_limits:
        The candidate may be executed only with narrower scope, lower risk, added
        checkpoints, additional tests, or explicit approval.

    defer:
        More information is needed before execution.

    reject:
        The candidate violates hard constraints or offers insufficient expected value.

    require_reframe:
        The target, constraints, input domain, or measurement method is too unclear to
        justify probing.

A rejected candidate should not simply disappear. It should be recorded as negative
knowledge with the reason for rejection.


============================================================
19. OBSERVATION
============================================================

Each probe must produce an observation record.

Observation record:

    - candidate id
    - actual input/action
    - timestamp or sequence id
    - environment assumptions
    - tool used
    - output observed
    - tests run
    - pass/fail result
    - measured score
    - unexpected side effects
    - validity status
    - uncertainty
    - raw evidence
    - link to trace/checkpoint

Do not discard failed results. Failed results may be the most valuable part of the search.


============================================================
19A. EVIDENCE LEDGER
============================================================

Every nontrivial probe should create an evidence ledger entry.

Evidence ledger entry:

    - claim being tested
    - candidate used
    - expected result
    - observed result
    - tests or checks performed
    - raw evidence location
    - interpretation
    - uncertainty remaining
    - whether the claim was supported, weakened, falsified, or left unresolved
    - whether the observation should affect future ranking

The evidence ledger prevents the agent from treating activity as progress.

A probe that produces no interpretable evidence should be marked as low value unless it
reveals a constraint, a measurement failure, or a need to reframe the problem.


============================================================
20. ANALYSIS
============================================================

After each observation, analyze:

    - Did the output move closer to the target?
    - Were hard constraints preserved?
    - Did the candidate reduce uncertainty?
    - Did the result support or weaken the inverse hypothesis?
    - Did it reveal a model mismatch?
    - Did it reveal an unsafe or low-value region?
    - Did it improve the best-known solution?
    - Was the result valid, noisy, suspicious, or artifactual?
    - Should the result be replicated?
    - Should the search continue, backtrack, reframe, or stop?


============================================================
21. NEGATIVE KNOWLEDGE
============================================================

Negative knowledge is structured memory of what did not work and why.

Record:

    - impossible regions
    - low-value regions
    - unsafe regions
    - high-cost low-gain regions
    - misleading assumptions
    - failed inverse hypotheses
    - failed search strategies
    - invalidated trials
    - model mismatches
    - measurement failures

Negative knowledge prevents repeated mistakes and improves future candidate ranking.


============================================================
21A. FAILURE TAXONOMY
============================================================

Failures should be categorized, not merely marked as failed.

Common failure types:

    target_failure:
        The candidate did not move the output toward the target.

    constraint_failure:
        The candidate violated a hard constraint.

    governance_failure:
        The candidate bypassed or weakened required approval, policy, or mediation.

    evidence_failure:
        The candidate produced no interpretable evidence.

    measurement_failure:
        The test, metric, trace, or observation method was invalid or too weak.

    model_failure:
        The active model predicted the wrong effect.

    complexity_failure:
        The candidate improved one score while making the system too large, coupled, or
        hard to replay.

    reversibility_failure:
        The candidate could not be safely rolled back or isolated.

    integration_failure:
        The candidate worked locally but conflicted with the larger architecture.

Each failure type should influence future candidate generation and ranking.


============================================================
22. BEST-KNOWN SOLUTION
============================================================

The best-known solution is not necessarily the true solution.

Always distinguish:

    best-found input:
        The best input found so far.

    local optimum:
        The best input found within a local region.

    global optimum:
        The best possible input across the full domain, only if supported by strong evidence.

    exact inverse:
        An input that produces the exact desired output.

    approximate inverse:
        An input that produces an output within tolerance.

    feasible solution:
        An input that satisfies all hard constraints.

    proven impossibility:
        Evidence that no acceptable input exists under current constraints.

Reports must state which status applies.


============================================================
23. BACKTRACKING
============================================================

Backtracking is required when:

    - the model is contradicted
    - the inverse hypothesis fails
    - the search strategy repeatedly fails
    - an apparatus or measurement fault is found
    - data processing is invalid
    - identifiability is insufficient
    - constraints were misunderstood
    - the target was poorly defined
    - a checkpoint is needed to restore a clean state

Backtracking must:

    - restore a compatible checkpoint
    - invalidate downstream state when appropriate
    - preserve audit history
    - record the cause
    - avoid hiding failed trials
    - avoid pretending the search was linear


============================================================
24. STOPPING CONDITIONS
============================================================

Stop when one of the following is true:

    - success is reached and verified
    - hard resource limits are reached
    - no candidate remains above the minimum expected value
    - marginal gain is below cost
    - further probing is unsafe
    - further probing is not justified by uncertainty reduction
    - the target is likely infeasible under current constraints
    - the problem must be reframed
    - the user or governing policy requires stopping

The termination reason must be explicit.

31. MINIMAL PSEUDOCODE
============================================================

PROCEDURE Inverse_Science_Method(problem_context):

    target = define_desired_output(problem_context)
    constraints = define_constraints(problem_context)
    input_domain = define_allowed_inputs(problem_context)

    IF target_is_invalid(target) THEN
        RETURN report_invalid_target()
    END IF

    IF constraints_are_violated_by_problem_definition(target, constraints) THEN
        RETURN report_constraint_failure()
    END IF

    black_box = characterize_black_box(problem_context, input_domain)

    identifiability = assess_identifiability(target, input_domain, black_box)

    IF identifiability == "not_identifiable" THEN
        RETURN report_unidentifiable_problem()
    END IF

    model_set = build_or_update_model_set(black_box, prior_observations = [])
    active_model = choose_provisional_model(model_set)

    inverse_hypothesis = form_inverse_hypothesis(
        target = target,
        constraints = constraints,
        input_domain = input_domain,
        model = active_model,
        black_box = black_box
    )

    candidate_pool = generate_candidates(
        inverse_hypothesis = inverse_hypothesis,
        input_domain = input_domain,
        constraints = constraints
    )

    WHILE stopping_condition_not_met():

        ranked_candidates = rank_candidates(
            candidate_pool = candidate_pool,
            target = target,
            constraints = constraints,
            model = active_model,
            negative_knowledge = negative_knowledge_base
        )

        IF ranked_candidates IS EMPTY THEN
            RETURN report_no_promising_candidates()
        END IF

        candidate = ranked_candidates[0]

        IF governance_allows(candidate) == false THEN
            retire_candidate(candidate)
            update_negative_knowledge(candidate, reason = "governance_blocked")
            CONTINUE
        END IF

        checkpoint_before_probe(candidate)

        observation = execute_probe(candidate)

        record_observation(candidate, observation)

        analysis = analyze_observation(
            target = target,
            candidate = candidate,
            observation = observation,
            model = active_model,
            constraints = constraints
        )

        update_best_known_solution(analysis)
        update_negative_knowledge(analysis)
        update_model_set(model_set, analysis)
        active_model = choose_provisional_model(model_set)
        inverse_hypothesis = refine_inverse_hypothesis(inverse_hypothesis, analysis)

        IF analysis.requires_backtracking THEN
            restore_compatible_checkpoint()
            invalidate_downstream_state()
            CONTINUE
        END IF

        IF analysis.suggests_new_candidates THEN
            candidate_pool = merge_candidate_pools(
                candidate_pool,
                generate_candidates_from_analysis(analysis)
            )
        END IF

        checkpoint_after_probe(candidate, observation, analysis)

    END WHILE

    RETURN final_report(
        target = target,
        constraints = constraints,
        best_known_solution = best_known_solution,
        negative_knowledge = negative_knowledge_base,
        uncertainty = current_uncertainty,
        termination_reason = termination_reason
    )

END PROCEDURE
